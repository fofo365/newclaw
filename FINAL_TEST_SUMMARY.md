## 📝 关于"其他四个通道"的问题

### OpenClaw 支持的通道

根据 `/root/.openclaw/openclaw.json` 配置，OpenClaw 支持以下通道：

| 通道 | OpenClaw | NewClaw | 状态 |
|------|----------|---------|------|
| **Feishu（飞书）** | ✅ | ✅ | 已迁移 |
| **WeCom（企业微信）** | ✅ | ✅ | 已迁移 |
| **QQ Bot** | ✅ | ❌ | 未迁移 |
| **DingTalk（钉钉）** | ✅ | ❌ | 未迁移 |
| **ADP OpenClaw** | ✅ | ❌ | 未迁移 |

### 为什么在状态确认里看不到？

**原因**:
1. NewClaw v0.4.0 专注于**飞书和企业微信**的集成
2. 其他通道（QQ、钉钉、ADP）仍在 OpenClaw 中运行
3. NewClaw 采用了**渐进式迁移策略**，先完成核心通道

### 当前架构

```
┌─────────────────────────────────────────┐
│           用户消息                       │
└────────────┬────────────────────────────┘
             │
       ┌─────▼─────┐
       │  路由层    │
       └─────┬─────┘
             │
    ┌────────┴────────┐
    │                 │
┌───▼──────┐    ┌────▼───┐
│ OpenClaw │    │ NewClaw │
│          │    │         │
│ • QQ Bot │    │ • Feishu│
│ • 钉钉   │    │ • WeCom │
│ • ADP    │    │         │
└──────────┘    └─────────┘
```

### 迁移计划

**短期** (v0.4.x - v0.5.0):
- 保持 OpenClaw 处理 QQ、钉钉、ADP
- NewClaw 专注飞书和企业微信
- 验证稳定性和性能

**中期** (v0.6.0 - v0.7.0):
- 迁移 QQ Bot 集成
- 迁移钉钉集成
- 统一通道接口

**长期** (v1.0.0):
- 完全替代 OpenClaw
- 单一架构，统一管理
